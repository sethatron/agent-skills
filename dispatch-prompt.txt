================================================================================
SKILL SPECIFICATION: dispatch
================================================================================

Use /skill-creator to implement the dispatch skill. This document is the
complete, authoritative specification. Follow every requirement exactly. Do
not invent requirements not stated here. Do not omit stated constraints.

Before implementing, internalize the following architectural reality:

  ┌─────────────────────── ARCHITECTURAL CONSTRAINT ───────────────────────┐
  │  A SKILL.md is a prompt loaded into Claude Code's context window. It is   │
  │  synchronous, session-scoped, and stateless between sessions. It cannot   │
  │  run autonomously, persist state across terminal sessions, or execute      │
  │  background tasks on its own.                                              │
  │                                                                            │
  │  The /dispatch workflow is inherently:                                     │
  │    • Long-running (spanning a full work day)                               │
  │    • Stateful across sessions (state must survive terminal closes)         │
  │    • Event-driven (Slack notifications on async triggers)                  │
  │    • Partially autonomous (cron-driven morning boot, EOD rollup)           │
  │                                                                            │
  │  A pure SKILL.md cannot address these requirements. The skill MUST         │
  │  scaffold a two-layer architecture:                                        │
  │    Layer 1 — SKILL.md: orchestration brain, workflow DSL, agent roles      │
  │    Layer 2 — Python scripts + state store: execution, persistence, hooks   │
  │                                                                            │
  │  The skill's first invocation must generate this full scaffolding. After   │
  │  scaffold generation, all subsequent invocations operate against the       │
  │  durable state layer — not just in-context memory.                         │
  └────────────────────────────────────────────────────────────────────────────┘

Apply current Claude Code best practices (Agent Teams via .claude/agents/,
hooks via .claude/hooks/, claude -p subprocess invocation for background
agents, SQLite for scalable state) wherever they improve on what is implied
in the requirements.

SECTION 2: ARCHITECTURE OVERVIEW
--------------------------------------------------------------------------------

2.1 Two-Layer Architecture

  LAYER 1 — SKILL.md (orchestration brain):
    - Defines the workflow DSL and step registry
    - Defines agent roles (Dispatch Coordinator, Optimus)
    - Defines sub-skill invocation contracts (/mr-review, /jira)
    - Defines the Slack notification schema
    - Defines Optimus triggers and reporting format

  LAYER 2 — Scripts and state (durable execution):
    - scripts/dispatch_runner.py   — main entrypoint, reads workflow.yaml,
                                     drives the step execution loop
    - scripts/state_store.py       — SQLite-backed task and session state
    - scripts/slack_notifier.py    — Slack MCP wrapper with message templates
    - scripts/optimus_runner.py    — spawns Optimus as a claude -p subprocess
    - scripts/log_writer.py        — structured log and artifact writer
    - scripts/cron_manager.py      — generates and manages cron job configs
    - scripts/bottleneck_detector.py — evaluates bottleneck conditions
    - scripts/check_env.py         — pre-validation (run first, always)

2.2 State Store Design

  The skill uses SQLite (not YAML/JSON) as the primary state store for
  scalability, queryability, and concurrent-safe writes:

    ~/.zsh/dispatch/dispatch.db

  Schema tables:
    tasks          — one row per task, with lifecycle state machine fields
    sessions       — one row per Claude Code session (opened/closed/compacted)
    step_log       — one row per workflow step execution with outcome
    bash_log       — one row per bash command executed during a session
    notifications  — one row per Slack message sent (dedup, audit)
    optimus_runs   — one row per Optimus agent invocation with output path
    bottlenecks    — one row per detected bottleneck, with resolution timestamp

  Human-readable artifacts are written to the filesystem (see Section 5).
  SQLite is the source of truth; filesystem artifacts are derived outputs.

2.3 Claude Code Agent Team Integration

  Optimus is implemented as a Claude Code agent (NOT an inline prompt):

    .claude/agents/optimus.md

  Optimus runs in its own context window, with no visibility into the dispatch
  session's context. It is invoked via:

    claude -p "$(cat ~/.zsh/dispatch/optimus_brief.md)" \
           --model claude-opus-4-6 \
           --allowedTools "Read,Bash(cat *),Bash(find *),Write" \
           --print > ~/.zsh/dispatch/{date}/optimus_report.md

  The optimus_brief.md is generated by dispatch_runner.py and contains:
  - Paths to all task logs for the period being reviewed
  - The current workflow.yaml
  - The current state of dispatch.db (exported as JSON summary)
  - The Optimus agent role definition
  - Explicit output format instructions

2.4 Hooks Integration

  Claude Code hooks (defined in .claude/hooks/) intercept tool calls and
  automatically log activity to dispatch.db without requiring explicit
  skill invocation at each step.

  Required hooks:
    PostToolUse/Bash   — log every bash command (command, cwd, exit_code,
                          duration_ms, timestamp) to bash_log table
    PostToolUse/Write  — log every file write (path, operation, timestamp)
    PostToolUse/Edit   — log every file edit (path, diff_lines, timestamp)
    PreToolUse/Bash    — check git guardrails (block add/commit/push unless
                          task has git_permission: true in state store)

  Hook scripts live in:
    .claude/hooks/post_bash.py
    .claude/hooks/post_write.py
    .claude/hooks/post_edit.py
    .claude/hooks/pre_bash_guard.py

  Hooks are generated by /dispatch scaffold and must be installed to
  .claude/hooks/ in the operator's home directory.

2.5 Session Continuity

  When a new Claude Code session begins and the dispatch skill is invoked:

  1. check_env.py validates the environment.
  2. state_store.py checks for an open session from today. If found:
       "Resuming session from <timestamp>. Last step: <step>. Status: <summary>"
  3. If the previous session was interrupted mid-step, the skill resumes from
     the last completed checkpoint, not from the beginning of the workflow.
  4. Each workflow step writes a checkpoint to the session table before and
     after execution. Mid-step interruption leaves the step in INTERRUPTED
     state, which triggers a resume prompt on next invocation.

SECTION 3: WORKFLOW DEFINITION (workflow.yaml)
--------------------------------------------------------------------------------

The operator's daily workflow is defined in a declarative YAML file:

  ~/.zsh/dispatch/workflow.yaml

This file is the single source of truth for what steps run, in what order,
with what parameters. The SKILL.md does not hardcode workflow steps; it reads
them from this file on every invocation. This is the mechanism that makes the
skill extensible without modifying the SKILL.md itself.

3.1 workflow.yaml Schema

  ---
  version: "1.0"
  operator:
    gitlab_username: "@zettatron"
    jira_username: "<operator-jira-email>"
    slack_channel: "#dispatch"
    slack_channel_id: "C0AQC48GL1G"
    timezone: "America/Phoenix"

  defaults:
    git_permission: false          # git add/commit/push blocked by default
    require_human_gate: true       # pause for operator confirmation at gates
    slack_notify_on: [step_complete, blocker_detected, bottleneck_detected,
                      task_closed, session_start, session_end, optimus_complete]

  steps:
    - id: mr_review_personal
      name: "Personal MR Review"
      skill: /mr-review
      args: "personal"
      description: "Review open MRs authored by operator"
      on_blocker: notify_slack
      timeout_minutes: 30
      tags: [gitlab, review, morning]
      enabled: true

    - id: mr_review_team
      name: "Team MR Review"
      skill: /mr-review
      args: "team"
      description: "Review open MRs from team members"
      on_blocker: notify_slack
      timeout_minutes: 60
      tags: [gitlab, review, morning]
      enabled: true

    - id: jira_mentions
      name: "Jira Mentions Check"
      skill: /jira
      args: "search assignee = currentUser() OR mentions = currentUser()"
      description: "Find Jira issues where operator is mentioned or assigned"
      on_new_mention: notify_slack
      timeout_minutes: 10
      tags: [jira, morning]
      enabled: true

    - id: bottleneck_scan
      name: "Bottleneck Detection"
      runner: scripts/bottleneck_detector.py
      description: "Identify cases where operator is blocking team progress"
      on_bottleneck_found: notify_slack
      timeout_minutes: 5
      tags: [analysis, morning]
      enabled: true

  human_gates:
    - after: mr_review_personal
      message: "Personal MR review complete. Ready to proceed to team review?"
    - after: jira_mentions
      message: "Jira check complete. Proceed with task prioritization?"

  schedule:
    morning_boot:
      cron: "0 8 * * 1-5"           # 8am weekdays
      command: "dispatch_runner.py --mode=morning"
      approval_required: true
    eod_summary:
      cron: "0 17 * * 1-5"          # 5pm weekdays
      command: "dispatch_runner.py --mode=eod"
      approval_required: true
    optimus_nightly:
      cron: "0 22 * * 1-5"          # 10pm weekdays
      command: "optimus_runner.py --period=day"
      approval_required: true

3.2 Workflow DSL Constraints

  - Steps with skill: invoke sub-skills by generating a structured handoff
    prompt and reading the resulting artifact (review.md, jira export, etc.)
    into the session log.
  - Steps with runner: invoke a Python script directly.
  - All steps must be idempotent. Re-running a step on the same day must
    detect a completed checkpoint and skip (not re-execute).
  - Steps may define on_* event handlers that map to slack notifications,
    human gate pauses, or Optimus triggers.
  - Adding a new step requires only adding an entry to workflow.yaml.
    No SKILL.md modification is required.

3.3 Git Permission Gate

  The workflow.yaml git_permission default is false. For any task where the
  operator explicitly grants git push/commit permission, they must run:

    /dispatch task git-allow <jira-id>

  This sets git_permission: true in the task record in dispatch.db. The
  pre_bash_guard.py hook checks this flag before allowing any git add,
  git commit, or git push command to execute. Attempting these commands
  without permission produces:

    BLOCKED: git add/commit/push requires explicit permission for task <id>.
    Grant with: /dispatch task git-allow <jira-id>

SECTION 4: TASK STATE MACHINE
--------------------------------------------------------------------------------

Every unit of work tracked by dispatch is a Task. Tasks have an explicit
lifecycle:

  PENDING → IN_PROGRESS → [BLOCKED] → IN_REVIEW → COMPLETE
                        → DEFERRED (carry to next day)
                        → ABANDONED

4.1 Task Record Schema (dispatch.db tasks table)

  task_id:           string (Jira issue key, e.g., PLAT-1234)
  title:             string
  description:       string (populated from Jira via /jira skill)
  status:            enum (PENDING, IN_PROGRESS, BLOCKED, IN_REVIEW,
                           COMPLETE, DEFERRED, ABANDONED)
  jira_status:       string (raw Jira status, synced from API)
  created_date:      ISO 8601 date (when task was first seen)
  started_at:        ISO 8601 timestamp (when operator began work)
  closed_at:         ISO 8601 timestamp
  deferred_to:       ISO 8601 date (for DEFERRED tasks)
  git_permission:    boolean (default false)
  priority:          integer (1-4, operator-assigned or derived from Jira)
  tags:              string[] (jira labels, gitlab labels, manual)
  blocker:           string (description of current blocker, if any)
  mr_links:          string[] (GitLab MR URLs associated with this task)
  related_tasks:     string[] (other task IDs)
  log_dir:           string (filesystem path to this task's artifact directory)
  optimus_reviewed:  boolean
  session_ids:       string[] (Claude Code session IDs that touched this task)

4.2 State Transitions

  PENDING → IN_PROGRESS:   /dispatch task start <jira-id>
  IN_PROGRESS → BLOCKED:   /dispatch task block <jira-id> --reason "<text>"
  BLOCKED → IN_PROGRESS:   /dispatch task unblock <jira-id>
  IN_PROGRESS → IN_REVIEW: /dispatch task submit <jira-id>
                            (triggers mr_review_personal step)
  IN_REVIEW → COMPLETE:    /dispatch task close <jira-id>
  any → DEFERRED:          /dispatch task defer <jira-id>
                            (writes carry-forward entry to next day's task.yaml)
  any → ABANDONED:          /dispatch task abandon <jira-id> --reason "<text>"

  All transitions are logged to step_log with timestamp, previous_status, and
  new_status. All transitions send a Slack notification unless suppressed with
  --quiet.

4.3 Multi-Day Task Carry-Forward

  Tasks in DEFERRED or IN_PROGRESS state at EOD are automatically written to
  the next business day's carry_forward.yaml:

    ~/.zsh/dispatch/{YYYY}/{MM}/{DD+1}/carry_forward.yaml

  The morning boot reads carry_forward.yaml and recreates those tasks in
  the new day's context with status preserved. A Slack message is sent at
  morning boot summarizing carried-over tasks:
    "Carried forward from yesterday: PLAT-123 (In Progress), PLAT-456 (Blocked)"

SECTION 6: BOTTLENECK DETECTION
--------------------------------------------------------------------------------

bottleneck_detector.py evaluates conditions that indicate the operator is a
blocking dependency. Each condition has an explicit detection rule, severity,
and Slack notification template.

6.1 Bottleneck Conditions

  CRITICAL (immediate Slack alert):
    - Team MR awaiting operator review for > 4 hours during business hours
    - Jira ticket assigned to operator with a comment from a teammate asking
      for status, with no response from operator in > 2 hours
    - Pipeline failure on operator's MR with no activity for > 1 hour

  HIGH (Slack alert at next natural pause in workflow):
    - Team MR in OPEN state with 0 reviews, created > 24 hours ago
    - Jira ticket in IN_REVIEW status assigned to operator for > 8 hours
    - Jira ticket blocking another ticket, where blocking ticket is OPEN

  MEDIUM (included in EOD summary):
    - Operator has > 5 open MR reviews outstanding
    - Operator has > 3 Jira tickets in IN_PROGRESS simultaneously
    - Any task DEFERRED more than twice

6.2 Bottleneck Record Schema (dispatch.db bottlenecks table)

  bottleneck_id:       uuid
  detected_at:        ISO 8601
  severity:           CRITICAL | HIGH | MEDIUM
  type:               string (mr_review_delay, jira_comment_pending, etc.)
  resource_id:        string (MR URL or Jira issue key)
  description:        string
  slack_notified:     boolean
  resolved_at:        ISO 8601 or null
  resolution:         string or null

SECTION 7: SUB-SKILL INVOCATION CONTRACT
--------------------------------------------------------------------------------

Dispatch orchestrates /mr-review and /jira as sub-skills. The invocation
contract defines how dispatch hands off, collects artifacts, and records
outcomes without losing sub-skill output.

7.1 Sub-Skill Invocation Pattern

  When dispatch runs a step with skill: defined in workflow.yaml:

  Step 1 — Generate handoff context:
    dispatch_runner.py writes a handoff.md to the step's working directory
    containing: the step parameters, the operator's current task context,
    and any relevant state from dispatch.db.

  Step 2 — Surface the sub-skill invocation prompt:
    The SKILL.md surfaces a clearly formatted invocation block:
      ╔═ DISPATCH: Invoking /mr-review personal ══════════════════╗
      ║  Task context: PLAT-1234                                   ║
      ║  Output dir: ~/.zsh/review/2026/04/01/                     ║
      ║  Started: 08:15:32                                         ║
      ╚══════════════════════════════════════════════════════════╝
    The operator sees this and the sub-skill runs within the same session.

  Step 3 — Collect artifacts:
    After the sub-skill completes, dispatch_runner.py:
    - Reads the sub-skill's output artifact (review.md, jira export)
    - Extracts the YAML frontmatter (both /mr-review and /jira produce
      machine-readable frontmatter — see Section 11 on dispatch ↔ sibling
      skill contracts)
    - Records the outcome in step_log
    - Symlinks the artifact into the task's directory in dispatch.db

  Step 4 — Post-step actions:
    - Send Slack notification if configured on step
    - Evaluate bottleneck conditions on the new data
    - Check for human gate

7.2 JIRA_CALLER Contract

  When dispatch invokes /jira for the jira_mentions step:
    JIRA_CALLER=dispatch

  This enforces FORCED READ-ONLY MODE on the jira skill. Dispatch never
  triggers jira write operations. If the operator wants to update a Jira ticket
  after reviewing dispatch output, they invoke /jira directly.

7.3 Git Guardrail Propagation

  When dispatch invokes /mr-review, the sub-skill's git guardrails remain
  in effect. Dispatch adds no git permissions beyond what the operator has
  explicitly granted per task.

8.3 Notification Deduplication

  The notifications table in dispatch.db stores a content_hash for each
  message. Before sending any Slack notification, slack_notifier.py checks
  that the same hash has not been sent in the past 30 minutes. This prevents
  spam during rapid step iteration or retry scenarios.

8.4 Cron Job Notification Protocol

  Whenever the skill identifies a useful cron schedule (from bottleneck
  patterns, Optimus suggestions, or explicit request):
  1. The cron config is written to ~/.zsh/dispatch/cron/ with status: PENDING.
  2. A Slack message is sent using cron_suggestion.md.j2.
  3. Operator must explicitly approve with: /dispatch cron approve <cron_id>
  4. Only after approval does cron_manager.py install the entry to crontab.
  5. All installed cron jobs are tracked in dispatch.db with their approval
     timestamp, rationale, and current enabled status.

SECTION 9: OPTIMUS AGENT
--------------------------------------------------------------------------------

Optimus is the "Optimization Agent" — a specialized Claude Code subagent
defined in .claude/agents/optimus.md. It runs in an isolated context window
with no access to the dispatch session, ensuring its analysis is unbiased.

9.1 Agent Definition (.claude/agents/optimus.md)

  The generated SKILL.md must produce the full content of this agent file.
  Key fields:
    name: optimus
    description: Reviews dispatch task logs and session data to identify
                 process improvements, tooling gaps, automation opportunities,
                 and workflow optimizations. Triggered after task closure or
                 by explicit /dispatch optimus invocation.
    model: claude-opus-4-6
    tools: Read, Bash(cat *), Bash(find *), Bash(python3 *), Write

9.2 Optimus Invocation Triggers

  Automatic (after task close):
    - When a task transitions to COMPLETE, optimus_runner.py checks if all
      tasks for the day are complete. If yes, Optimus is invoked for the day.
    - If not all tasks are complete, Optimus is queued and runs at EOD.

  Manual (operator-forced):
    - /dispatch optimus: runs Optimus on all tasks in the current period,
      including incomplete tasks.
    - /dispatch optimus --period=week: runs Optimus across the current week's
      task and session logs.

9.3 Optimus Brief (optimus_brief.md)

  optimus_runner.py generates this file before invoking Optimus. Contents:

  ## Review Period
  - Date range: {{ start_date }} to {{ end_date }}
  - Total tasks: {{ task_count }}
  - Completed: {{ completed }} | Deferred: {{ deferred }} | Abandoned: {{ abandoned }}

  ## Task Log Paths
  {{ task_log_paths }}  (list of paths for Optimus to read)

  ## Session Data (JSON summary from dispatch.db)
  {{ session_summary_json }}

  ## Workflow Configuration
  {{ workflow_yaml_contents }}

  ## Optimus Instructions
  You are Optimus, the workflow optimization agent for @zettatron's DevOps
  dispatch system. You have no access to the operator's current session.
  Your role is to perform a critical, data-driven analysis of the task logs,
  bash command history, bottleneck records, and session metadata provided.

  Produce an optimization report in the format defined in Section 9.4.
  Ground every recommendation in specific evidence from the logs.
  Do not hallucinate patterns not present in the data.
  Label inferences as [INFERENCE] and observations as [OBSERVED].

9.4 Optimus Report Format (optimus_report.md)

  ---
  period_start: <date>
  period_end: <date>
  tasks_reviewed: <integer>
  total_findings: <integer>
  findings_by_category:
    tooling: <integer>
    process: <integer>
    automation: <integer>
    mcp_integration: <integer>
    workflow_gap: <integer>
  generated_at: <ISO 8601>
  ---

  ## Executive Summary
  (2-3 sentences: what period covered, top 3 findings, overall assessment)

  ## Observed Patterns
  (Data-driven observations from log evidence — no speculation)

  ## Findings

  ### [FINDING-001] <Title>
  Category: tooling | process | automation | mcp_integration | workflow_gap
  Severity: HIGH | MEDIUM | LOW
  Evidence: (specific log entries, command patterns, timestamps)
  Recommendation: (concrete, actionable change)

SECTION 14: CONTEXT WINDOW MANAGEMENT
--------------------------------------------------------------------------------

A full day's worth of task logs, bash command history, and sub-skill artifacts
can easily exceed Claude Code's context window. The SKILL.md must define an
explicit strategy for managing context across a multi-step day.

14.1 Progressive Loading

  - At session start, dispatch loads only: session.yaml, task.yaml (today),
    carry_forward.yaml, and the relevant workflow.yaml steps.
  - Sub-skill artifacts (review.md, jira exports) are summarized on load,
    not fully expanded. The full artifact path is recorded; the content is
    loaded on demand only when a step requires it.

14.2 Compaction Triggers

  - The dispatch SKILL.md instructs Claude Code to use /compact at defined
    logical breakpoints: after each workflow step completes, before invoking
    a sub-skill, before invoking Optimus.
  - Compaction points are logged to session.yaml as context_compaction events.
  - After compaction, dispatch_runner.py re-hydrates state from dispatch.db
    (not from context memory), ensuring continuity.

14.3 Optimus Context Isolation

  Optimus runs as a subprocess (claude -p) with its own context window. It
  receives only the optimus_brief.md and agent definition — it does not inherit
  the dispatch session context. This is by design: a fresh context produces
  more objective analysis.

14.4 Large Log Handling

  If a task's bash_commands.log exceeds 500 lines, dispatch_runner.py
  generates a summarized version (bash_commands_summary.log) using frequency
  analysis and de-duplication. Optimus receives the summary; the full log is
  preserved on disk. The threshold is configurable in dispatch.yaml.

SECTION 15: PRIORITY AND TRIAGE FRAMEWORK
--------------------------------------------------------------------------------

Dispatch operates across two systems (GitLab, Jira) that each have their own
priority signals. The skill must synthesize these into a unified priority view.

Priority scoring (computed by bottleneck_detector.py at session start):

  P1 — CRITICAL: address immediately
    - MR with failing pipeline, author waiting for review
    - Jira ticket in BLOCKED state where operator is the blocker
    - Teammate's MR open > 4 hours with 0 reviews from operator

  P2 — HIGH: address before end of day
    - Personal MR with review comments requiring response
    - Jira ticket with unread @mentions from team lead or PM
    - Team MR open > 24 hours

  P3 — NORMAL: address within 2 days
    - Team MR in draft state (awareness, not action required)
    - Jira ticket in TO DO status assigned to operator
    - Jira tickets where operator is CC'd

  P4 — LOW: address when convenient
    - Informational Jira comments without action required
    - Team MRs with active reviews already in progress from others

  At session start, dispatch_runner.py generates a prioritized work queue
  and presents it to the operator before beginning the workflow steps:
    "Today's priority queue:
     [P1] Review PLAT-123 MR (teammate blocked)
     [P2] Respond to review comments on branch feature/xyz
     [P3] 3 team MRs awaiting review
     Proceed with morning workflow? [Y/n]"

SECTION 16: FAILURE HANDLING AND RECOVERY
--------------------------------------------------------------------------------

16.1 Step Failure Protocol

  If a workflow step fails (sub-skill error, API timeout, hook exception):
  1. The failure is recorded in step_log with error details.
  2. A Slack notification is sent to #dispatch.
  3. The step is marked FAILED in session.yaml.
  4. The workflow continues to the next step (unless the failed step is marked
     blocking: true in workflow.yaml, in which case the operator is prompted).
  5. Failed steps are retried once automatically on next invocation.
  6. After 3 consecutive failures of the same step, the step is disabled until
     the operator manually re-enables it.

16.2 Sub-Skill Failure Protocol

  If /mr-review or /jira fail to produce an artifact:
  - Dispatch records the failure and skips artifact collection for that step.
  - The bottleneck detector does not run on missing data.
  - A clear error is surfaced: "Step mr_review_personal produced no artifact.
    Check /mr-review skill configuration. Continuing to next step."

16.3 Slack MCP Unavailable

  If the Slack MCP is unavailable:
  - All pending notifications are queued in dispatch.db (notifications table,
    status: PENDING).
  - At the next successful Slack connection, queued notifications are sent in
    chronological order (no dedup bypass — duplicates are still suppressed).
  - A local fallback log is written: ~/.zsh/dispatch/{date}/slack_queue.yaml

16.4 Database Corruption

  If dispatch.db is corrupt or unreadable:
  1. Attempt SQLite integrity check: PRAGMA integrity_check.
  2. If corrupt: create dispatch.db.backup.<timestamp>, re-initialize schema.
  3. Attempt to recover task records from filesystem (task.yaml files).
  4. Surface a detailed recovery report to the operator.
  5. Send Slack notification: "dispatch.db was corrupted and re-initialized.
     Manual review of ~/.zsh/dispatch/ recommended."
