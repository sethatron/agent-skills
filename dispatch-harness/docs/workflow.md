# dispatch-harness Workflow Integration

## DSI Type

dispatch-harness is TYPE B (on-demand) with one TYPE C element: the telemetry
pre-Optimus hook triggered during dispatch's EOD workflow.

## Telemetry Pre-Optimus Hook

Before Optimus runs (in dispatch's `run_eod_workflow()`), generate the telemetry
digest so Optimus can consume it:

```python
# In dispatch_runner.py run_eod_workflow(), before Optimus invocation:
from pathlib import Path
import subprocess

digest_script = Path.home() / "agent-skills" / "dispatch-harness" / "scripts" / "telemetry_builder.py"
if digest_script.exists():
    subprocess.run(["python3", str(digest_script)], timeout=60)
```

This is defined in `references/dispatch-changes.md` and applied during init.

## No Workflow Steps

dispatch-harness does not define workflow.yaml steps. It is invoked on-demand
via `/dispatch-harness` subcommands. The only automated trigger is the
telemetry digest generation before Optimus, which is a hook in dispatch's
EOD workflow, not a standalone workflow step.

## Entropy Agent Schedule

The entropy agent runs monthly on explicit operator invocation only:
`/dispatch-harness entropy`

It is never a workflow step or cron job. Monthly cadence is a guideline,
not an automated schedule.

## Integration with dispatch Weekly Report

dispatch's weekly_report.md gains a Framework Quality Grades section:

```markdown
## Framework Quality Grades
| Skill | Overall | SKILL.md | Scripts | docs/ | Contracts |

```

This is generated by `grade_reporter.py` (currently a stub) and inserted
by dispatch_runner.py when generating weekly reports.
